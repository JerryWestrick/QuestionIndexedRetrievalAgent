# QIRA Design

## Two-Phase Architecture

### Runtime (the product)
Fixed. Two tools (`qira_search`, `qira_read`) operating against a defined storage contract. Build once. This is what the LLM interacts with — it never knows or cares how the data got there.

### Preparation (adapters)
Variable. Each source format (markdown, PDF, docx, HTML, etc.) needs its own pipeline to ingest documents and produce storage that fulfills the runtime contract. These are adapters — examples/templates, because each deployment's actual documents dictate the preparation stack.

```
  Markdown ──► ┐
  PDF      ──► ├── Preparation Adapter ──► Corpus Directory ──► Runtime
  Docx     ──► ┘                              (fixed)         (fixed)
  Custom   ──►
```

### Build Order
1. Define the storage contract (what runtime needs)
2. Build the runtime (`qira_search`, `qira_read`)
3. Build preparation adapters one at a time (markdown first — cleanest source, easiest to validate)

## KePrompt Integration

### Directory Structure
```
prompts/functions/
  qira                        # Function executable
  qira.prompt                  # Generated by --initialize
  qira-corpus/
    hr-regulations/
      hr-regulations.db            # SQLite — pre-formatted section store
      chroma/                      # ChromaDB — question vector index
      corpus.md                    # Corpus identity: name, description, embedding config, worked example
    textual-python/
      textual-python.db
      chroma/
      corpus.md
```

Each corpus = a directory. Self-contained. Add a corpus = drop in a directory. Remove = delete it.

### qira Executable — Two Operational Contexts

**Runtime** (called by KePrompt during conversations):
- `--list-functions` → returns `[qira_search, qira_read]` in OpenAI function calling format (standard KePrompt contract)
- `qira_search` → JSON on stdin, executes search, returns markdown to stdout
- `qira_read` → JSON on stdin, fetches section, returns markdown to stdout

**Release** (called by the engineer when corpora are added, removed, or updated):
- `--initialize` → scans corpus directories, reads each `corpus.md`, generates `qira.prompt`

The runtime modes are the product — what the LLM uses. `--initialize` is a maintenance tool that keeps `qira.prompt` in sync with the current set of corpora.

### Generated qira.prompt
Contains:
- QIRA retrieval instructions (reformulate question, use corpus filtering, browse results, read sections, follow cross-references)
- Corpus catalog listing each available corpus with name and description

### KePrompt Prompt Usage
```
.prompt "name":"MyApp", "version":"1.0"
.functions qira
.include functions/qira.prompt
.user <<question>>
.exec
```

## Section IDs

IDs are globally unique, prefixed with corpus: `{corpus}:{hierarchy}`

Example: `hr-regulations:3.2.4`

- Unique across the entire QIRA instance, not just within a corpus
- The LLM sees these in search results, copies them directly into `qira_read` calls
- One string, no assembly — `qira_read("hr-regulations:3.2.4")`
- LLM can still derive navigation from the hierarchy portion: parent is `hr-regulations:3.2`, sibling is `hr-regulations:3.2.5`
- The corpus prefix tells the runtime which directory to look in
- **Cross-references in content** are rewritten by the preparation adapter to full IDs (e.g. "see section 3.1.2" becomes "see section hr-regulations:3.1.2"). The LLM can act on them immediately without inferring the corpus. One-time preparation cost, zero runtime ambiguity.

## Storage Contract

The runtime requires this to exist per corpus directory. How it gets populated is the adapter's problem.

### Section Store ({corpus}.db — SQLite)
Pre-formatted at preparation time. RA serves `read_entry` verbatim; for search results, RA adds the heading with distance at runtime.

| Field | Purpose |
|-------|---------|
| `id` | Full corpus-prefixed section ID (e.g. `hr-regulations:3.2.4`) — lookup key |
| `title` | Section title (e.g. `Annual Allotment`) — used by RA to build search result heading |
| `search_entry` | Pre-formatted markdown body (breadcrumb, questions, excerpt) — RA prepends `## -{distance}- {id} {title}` at runtime |
| `read_entry` | Pre-formatted markdown for read (h1, breadcrumb, full content, subsections) — served verbatim |

### Question Index (chroma/ — ChromaDB)
Each entry is a generated question linked to a section:

| Field | Purpose |
|-------|---------|
| `question` | Generated question text |
| `embedding` | Vector embedding of the question |
| `section_id` | Back-reference to section store (e.g. `3.2.4`) |

Corpus filtering is implicit — each corpus has its own ChromaDB. ChromaDB returns section IDs; all formatting comes from SQLite.

### Corpus Identity (corpus.md)
Each corpus directory includes a `corpus.md` with name, description, embedding config, and a worked example. `--initialize` reads these to generate `qira.prompt`. See qi-ra-interface.md for the full spec.

## Runtime Implementation

### `qira_search(corpus, question, n_results=5)`
1. Parse corpus from input
2. Open `qira-corpus/{corpus}/chroma/`
3. Query ChromaDB: `collection.query(query_texts=[question], n_results=n_results, include=["metadatas", "distances"])` — ChromaDB embeds the query internally using the embedding function QI stored at index time. RA never touches vectors.
4. Deduplicate: group by section ID, keep the best (lowest) distance per section
5. Fetch `title` and `search_entry` for each from `qira-corpus/{corpus}/{corpus}.db`
6. Assemble each result: `## -{distance}- {id} {title}\n{search_entry}`
7. Concatenate and return markdown. If no results, return `## Nothing Found\n`

### `qira_read(section_id)`
`section_id` format: `{corpus}:{hierarchy}` (e.g. `hr-regulations:3.2.4`)
1. Split into corpus + hierarchy
2. Open `qira-corpus/{corpus}/{corpus}.db`
3. Fetch `read_entry` by hierarchy
4. Return markdown

### Error Handling

Follows the KePrompt external function protocol. All errors go to stderr with non-zero exit. KePrompt catches them and delivers the error text to the LLM as a `tool_result`, so the LLM can decide to retry, adjust, or report.

**Successful outcomes** (stdout, exit 0):
- Search with results → concatenated `search_entry` markdown
- Search with no results → `## Nothing Found\n` (not an error — the corpus just doesn't cover it)
- Read → `read_entry` markdown

**Errors** (stderr, exit 1):
- Unknown corpus → `Error: unknown corpus '{name}'. Available: {list}`
- Section not found → `Error: section '{id}' not found`
- Corrupt/missing DB → `Error: failed to open corpus '{name}': {detail}`
- Bad input JSON → `Error: invalid input: {detail}`

## Preparation Adapters

Each adapter must produce a corpus directory (`{corpus}.db` + `chroma/`) from source documents.

### Format-specific (steps 1-3)
1. Parse source document into a section tree
2. Assign hierarchical IDs
3. Extract titles, breadcrumbs, content
4. Determine readable units — the leaf depth at which sections are self-contained enough for LLM comprehension. Deeper levels become inline content within their parent's `read_entry`, not separate sections. Questions are still generated from sub-sections but point to the parent as the readable unit.

### Shared infrastructure (steps 4-6)
4. Generate questions per section (LLM call)
5. Pre-format `search_entry` and `read_entry` markdown per section
6. Vectorize questions and store in ChromaDB
7. Store sections in SQLite

### Adapter priority
1. **Markdown** — clean heading structure maps directly to section tree. Build and validate first.
2. **PDF** — requires structure extraction (headings, TOC). Harder, common need.
3. **Docx** — heading styles map to hierarchy. Similar to markdown.
4. **HTML** — `<h1>`-`<h6>` map to hierarchy. Similar to markdown.
5. **Custom** — whatever the application needs.
